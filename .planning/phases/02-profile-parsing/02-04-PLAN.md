<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 1: Validate selectors against live LinkedIn DOM</name>
  <what-built>
    A complete profile parsing pipeline: selectors.ts defines all CSS selector strings, parser.ts extracts name/headline/about/skills/experience/education from a Document, content.ts runs extraction on page load and on SPA navigation, background.ts stores the result. All unit tests pass in Vitest.
  </what-built>
  <how-to-verify>
    Step 1 — Build and load the extension in Chrome:
    - Run: `cd linkedin-hhrh-screener && pnpm build`
    - In Chrome: Settings → Extensions → Load unpacked → select `linkedin-hhrh-screener/.output/chrome-mv3`

    Step 2 — Open a LinkedIn profile page (any real profile you have access to).

    Step 3 — Validate each selector in Chrome DevTools Console (F12 → Console tab).
    Run each line and confirm it returns the expected elements (not null, not empty NodeList):

    ```javascript
    // Name
    document.querySelector('h1')?.textContent?.trim()

    // Headline
    document.querySelector('div.text-body-medium.break-words')?.textContent?.trim()

    // About
    document.querySelector('section[data-view-name="profile-card"] div.inline-show-more-text span[aria-hidden="true"]')?.textContent?.trim()

    // Skills (scroll to Skills section first — must be in viewport)
    document.querySelectorAll('a[data-field="skill_card_skill_topic"] span[aria-hidden="true"]').length

    // Experience section exists
    document.querySelector('#experience')

    // First experience title (scoped to #experience)
    document.querySelector('#experience').querySelector('div.t-bold span[aria-hidden="true"]')?.textContent?.trim()

    // Education section exists
    document.querySelector('#education')
    ```

    Step 4 — Check browser console for PROFILE_PARSED log output:
    - Open DevTools → Background service worker (click "service worker" link in chrome://extensions)
    - Look for: `[HHRH] Profile parsed: <candidate name> | Health ok: true`

    Step 5 — Test SPA navigation:
    - From the current profile, click to another LinkedIn profile (do not reload the page — use LinkedIn's own navigation links)
    - Confirm you see a second `[HHRH] Profile parsed:` log in the background console with the new candidate's name
    - Confirm only ONE log per navigation (debounce working — not 2-3 duplicate messages)

    Step 6 — If any selector returns null or empty results:
    - Right-click the relevant element on the page → Inspect
    - Find the correct selector in the current DOM
    - Update that selector string in `linkedin-hhrh-screener/src/parser/selectors.ts`
    - Re-run `pnpm build` and reload the extension
    - Re-run the DevTools check to confirm the updated selector works
    - Run `pnpm vitest run` — update fixture HTML in parser.test.ts to match the corrected selector
  </how-to-verify>
  <resume-signal>
    Type "approved" if all selectors returned real data and SPA navigation triggered re-extraction.
    Type "selectors-updated" if you had to correct one or more selectors in selectors.ts (describe which ones changed so the summary captures the change).
    Type "blocked" if LinkedIn's DOM has changed significantly and extraction returns mostly empty — describe what you see.
  </resume-signal>
</task>
